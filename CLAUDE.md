# Leonardo School

## Architecture Overview
**Next.js 16 (App Router) + React 19** + **Firebase Auth/Storage/FCM** + **PostgreSQL/Prisma 7** + **tRPC 11** + **TanStack Query** + **Tailwind 3**. Package manager: **pnpm**.

> A separate **Expo / React Native** app (`leonardoschool-mobile`) lives under `mobile/`. Everything below — commands, conventions, complexity rules — targets the **Next.js web app**. Don't apply web conventions to `mobile/`, and exclude it when searching for web code.

### Route Groups Structure
- `app/(marketing)/` → Public pages with Header/Footer (SEO-optimized)
- `app/(app)/` → Protected pages with unified routes (role-based content rendered via components, not separate routes per role). Page folders are in Italian (`simulazioni`, `domande`, `studenti`, `contratti`, ...)
- `app/auth/` → Standalone auth pages (login, register, complete-profile)
- `app/api/` → Route handlers (notably `app/api/auth/me/route.ts` for the DB sync)
- `proxy.ts` → Edge middleware for auth & role-based routing (this is the project's `middleware.ts` equivalent — **NOT** `middleware.ts`)

## Essential Commands
```bash
pnpm dev                 # Start dev server with Turbopack (localhost:3000)
pnpm dev:lan             # Dev server bound to 0.0.0.0 (test on LAN devices)
pnpm build               # prisma generate + next build
pnpm lint                # ESLint (includes eslint-plugin-sonarjs) — see note below
pnpm tsc --noEmit        # TypeScript check — run after every change, zero errors required

pnpm prisma:generate     # REQUIRED after every schema change
pnpm prisma:studio       # DB GUI (localhost:5555)
pnpm prisma:migrate      # Create/apply a dev migration
pnpm prisma:push         # Push schema without a migration
pnpm prisma:reset        # Reset the dev database (destructive)
pnpm prisma:seed         # Seed the database
pnpm cleanup:dry         # Preview cleanup of stale data (see docs/CLEANUP.md)
```

> **`pnpm lint` is currently broken locally**: it fails with `Cannot find module '@babel/core'` (`eslint-plugin-react-hooks` can't resolve it under pnpm's layout, even though `@babel/core` is installed). Lint runs correctly in CI. Locally, rely on `pnpm tsc --noEmit` and let CI report lint — don't try to "fix" the env by adding packages.

### Environment Switching
The active `.env` is generated by copying one of `env/.env.{local,test,production}`. Never hand-edit `.env` for an env change — switch instead:
```bash
pnpm env:local     # Local Docker Postgres (DATABASE_URL on localhost:5433)
pnpm env:test      # Neon test DB
pnpm env:prod      # Production
pnpm env:current   # Show which env is active
```

Tests live in `tests/` (not co-located): `unit/`, `integration/`, `e2e/`, with shared `mocks/` (Firebase, Prisma) and `setup.ts`. Mirror the source tree under `tests/unit/`. See `docs/TESTING.md` for full conventions.

## Authentication Flow
1. Client calls `firebaseAuth.login/register()` from `lib/firebase/auth.ts`
2. **MUST** then call `/api/auth/me` (POST) to sync the user into PostgreSQL
3. Server sets cookies: `auth-token` (HttpOnly), `user-role`, `profile-completed`
4. Students/Collaborators are redirected to `/auth/complete-profile` if `profileCompleted=false`
5. Account is active only when `profileCompleted=true` **AND** `isActive=true` (admin approval)

## tRPC Patterns
All API logic in `server/trpc/routers/`. Register new routers in `server/trpc/routers/index.ts`. Procedures from `server/trpc/init.ts`:
- `publicProcedure` — no auth
- `protectedProcedure` — authenticated
- `adminProcedure` — ADMIN only
- `collaboratorProcedure` — COLLABORATOR only
- `staffProcedure` — ADMIN or COLLABORATOR
- `studentProcedure` — STUDENT only

Every procedure is wrapped with `withRequestContext` (AsyncLocalStorage, akin to Spring's `RequestContextHolder`) — request-scoped data flows through `lib/utils/requestContext.ts`, so don't thread request metadata manually. Use `ctx.prisma.$transaction()` for atomic operations.

## Database (Prisma)
Schema: `prisma/schema.prisma`. **UserRole**: `ADMIN`, `COLLABORATOR`, `STUDENT`. Subjects are dynamic DB rows (`CustomSubject`) with per-subject colors. Run `pnpm prisma:generate` after **every** schema change.

## Color System (MANDATORY)
Never hardcode hex values. Use `lib/theme/colors.ts`:
```typescript
import { colors, generateDynamicSubjectColor } from '@/lib/theme/colors';
<div className={colors.primary.gradient} />              // brand
const c = generateDynamicSubjectColor(subject.color);    // dynamic subject color from DB
```
Categories: `colors.primary.*`, `colors.background.*`, `colors.status.*`. If you need a color that isn't there, add it to `colors.ts` following existing patterns — never inline `bg-[#...]`. Support both light and dark mode (avoid raw `bg-white`/`text-black`).

## Standard UI/Feature Conventions
- **Loaders**: use `components/ui/loaders` — never custom spinners or "Loading..." text. `Spinner`, `PageLoader` (for `loading.tsx`), `ButtonLoader` (wraps button content during mutations), `DotsLoader`, `Skeleton*`.
- **Mutations**: always add `onError: handleMutationError`, destructured from the `useApiError()` hook (`lib/hooks/useApiError`): `const { handleMutationError } = useApiError();`. Success feedback via `showSuccess(title, message)` from `useToast` (`components/ui/Toast`). `parseError()` (`lib/utils/errorHandler`) converts errors to Italian.
- **Validation**: reuse `lib/validations/*` (`authValidation`, `profileValidation`, etc.) and `lib/utils/stringUtils` (`normalizeName`), `lib/utils/sanitizeHtml` for HTML.
- **Email**: server-side templates in `lib/email/`.

## Code Quality & Complexity Standards
- Max cyclomatic complexity per function/component: **15**
- Max lines: component file **400**, tRPC router file **500** (split into co-located `[router].helpers.ts`)
- Max **8** `useState` and **5** `useMemo`/`useCallback` per component — extract a custom hook if exceeded
- Render branch >50 lines → named sub-component; logic reused 2+ times → custom hook in `lib/hooks/` (`use[Feature]Data.ts` for tRPC, `use[Component]State.ts` for UI); pure computation >10 lines reused → `lib/utils/`
- Modal/form content → dedicated `XxxFormModal.tsx` file
- tRPC business logic >20 lines → pure, unit-testable `[router].helpers.ts`
- Naming: Components `PascalCase.tsx`, hooks `useCamelCase.ts`, helpers `camelCase.helpers.ts`, types `camelCase.types.ts`
- Modern React: `useOptimistic` over manual rollback; `startTransition` for non-urgent updates; avoid prop drilling beyond 2 levels

## Claude-Specific Rules
- When reducing complexity, extract pure helpers first, then hooks, then sub-components
- Keep public APIs unchanged: tRPC procedure names, component props, hook return shapes
- Run `pnpm tsc --noEmit` after every file change — zero errors required
- Never add new npm packages without explicit user approval
- Prefer editing existing files over creating new ones
- Code, folders, and comments in English; **all user-facing strings in Italian**
- Comments explain WHY (non-obvious constraint/workaround), never WHAT

## Common Mistakes to Avoid
1. Missing DB sync: after Firebase auth, always POST `/api/auth/me`
2. Stale Prisma client: run `pnpm prisma:generate` after schema changes
3. Hardcoded colors: always use `colors.ts`
4. Wrong middleware file: use `proxy.ts`, NOT `middleware.ts`
5. Hand-editing `.env`: switch with `pnpm env:*` instead
6. Hardcoded Italian strings in English code paths: user messages stay Italian, code stays English
